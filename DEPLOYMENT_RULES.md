# GitHub Pages 部署安全与命名规范

本仓库（jd.git）是**唯一的**公网岗位卡发布渠道。所有暴露到公网（GitHub Pages）的文件必须遵守以下红线规则。

## 🔴 安全审核红线（部署前必做）

部署前，必须运行以下 `grep` 命令检查是否有内部情报泄露：

```bash
grep -rni "客户真名\|创始人真名\|费率\|佣金\|AUM\|预算\|融资额" --include="*.html" .
```

检查项目包括但不限于：
1. **公司名**：所有客户公司名称必须化名或使用特征描述（如"头部高频量化自营"）。禁止出现客户的营业执照名称、招聘常用名或关联公司名（如 Galxe）。
2. **人名**：禁止出现创始人、高管或面试官的真实全名。只能用"创始人 / Co-founder"或"联合创始人 W"等代称。
3. **资金与商业机密**：禁止出现确切的管理规模（AUM，可说"百亿级"不能说"120亿"）、提成费率（如 30%）、固定猎头费、预算上限。
4. **内部沟通渠道**：禁止在代码中硬编码任何飞书文档链接、内部邮箱、内部测试服务器地址。

## 🟡 文件命名规范

**文件名是URL的一部分，候选人可见，因此文件名本身也必须匿名化。**

### ✅ 正确示例（特征命名法）
文件名应描述**岗位类型、技术栈或业务方向**。
- `rust-quant-dev.html` (推荐)
- `quant-pm-hft.html` (推荐)
- `trading-sys-engineer.html` (推荐)
- `candidate-profile-x3m9.html` (候选人画像加随机后缀)

### ❌ 错误示例（客户名命名法）
禁止使用以下方式命名文件：
- 客户中文名拼音：`jishang-pm.html`, `pengfu-job.html`
- 客户英文名：`monad-role.html`, `alva-creator.html`
- 客户简称或股票代码。

## 🟢 目录结构规范

- **单源真实（Single Source of Truth）**：本目录（`jd-deploy/`）是唯一应当具有 `.git` 的部署目录。不要在客户档案的各自目录下建立单独的 `岗位卡部署` git 仓库，以防历史冲突和操作踩脚。
- **备份策略**：不再需要的过期文件、含有真实公司名的废弃版本、或仅限内部查看的方案报告（如 BP），应保存在 `客户档案/客户X/` 的本地不上传文件夹中，**绝不要**放入本目录。
